## 3. 问题清单（按严重级别）

## P0（必须先修）

### P0-1 文档定位与仓库现状冲突（SSOT 失真）
- 严重性：P0
- 问题描述：文档声明自己是“唯一技术真相源”，但大量关键文件在仓库中并不存在。
- 证据：
  - `finance-news-radar/docs/开发文档.md:4`（声明 SSOT）
  - `finance-news-radar/docs/开发文档.md:109`（目录结构宣称存在核心文件）
  - 当前缺失：  
    - `finance-news-radar/scripts/update_finance.py`  
    - `finance-news-radar/.github/workflows/update-finance.yml`  
    - `finance-news-radar/README.md`  
    - `finance-news-radar/index.html`  
    - `finance-news-radar/requirements.txt`
- 风险影响：
  - 新开发者会误判“已实现范围”，导致错误排期和错误依赖。
  - 交付验收标准不清晰，容易出现“文档完成但功能未完成”。
- 修正建议：
  - 将文档拆分为“当前状态（As-Is）”和“目标设计（To-Be）”。
  - 将“SSOT”改为“设计草案 vX”或“实施规划文档”，避免误导。

### P0-2 数据源统计口径不一致
- 严重性：P0
- 问题描述：文档中的数据源数量与 OPML 实际配置不一致，且文档内部前后矛盾。
- 证据：
  - 文档写“总计 ~25 个”：`finance-news-radar/docs/开发文档.md:226`
  - 文档示例 `source_count: 30`：`finance-news-radar/docs/开发文档.md:513`
  - OPML 实际 `xmlUrl` 条目为 26 条：`finance-news-radar/feeds/finance.opml`
- 风险影响：
  - 统计面板、监控指标、测试断言会基于错误基线。
  - 开发者无法确定“预期采集规模”。
- 修正建议：
  - 统一以 `finance.opml` 自动统计为单一口径。
  - 文档中将“估算值”改为“动态值（由 OPML 实时计算）”。

## P1（建议开发前修）

### P1-1 同一接口参数前后不一致
- 严重性：P1
- 问题描述：表格、示例代码、描述中的参数值不一致。
- 证据：
  - 华尔街见闻：表格 `limit=20`，示例代码 `limit=50`  
    - `finance-news-radar/docs/开发文档.md:158`  
    - `finance-news-radar/docs/开发文档.md:257`
  - 财联社：表格 `rn=20`，示例代码 `rn=50`  
    - `finance-news-radar/docs/开发文档.md:174`  
    - `finance-news-radar/docs/开发文档.md:309`
- 风险影响：
  - 采集量、速率、数据质量评估不可比。
  - 开发和测试使用不同参数，结果无法复现。
- 修正建议：
  - 新增“参数基线表”（每个源仅一组权威参数）。
  - 代码段和表格引用同一参数常量，避免双份维护。

### P1-2 华尔街见闻 channel 说明与示例实现不一致
- 严重性：P1
- 问题描述：文档声明支持多个 channel，但示例代码仅覆盖部分 channel。
- 证据：
  - 说明列出多个 channel：`finance-news-radar/docs/开发文档.md:162`
  - 示例代码仅 `global/us-stock/a-stock`：`finance-news-radar/docs/开发文档.md:260`
- 风险影响：
  - 开发者不清楚是否需要全量覆盖，导致范围漂移。
- 修正建议：
  - 在文档明确“第一期必选 channel”和“可选 channel”。
  - 将实现范围写入 Phase 1 验收标准。

### P1-3 过滤逻辑存在潜在大小写归一化隐患
- 严重性：P1
- 问题描述：文本先 `lower()`，但关键词表混合大小写，若 `contains_any_keyword` 未统一归一化，存在漏判风险。
- 证据：
  - 文本 lower：`finance-news-radar/docs/开发文档.md:479`
  - 混合大小写关键词：`finance-news-radar/docs/开发文档.md:396`、`finance-news-radar/docs/开发文档.md:413`
- 风险影响：
  - 英文关键词召回率下降，导致金融内容误过滤。
- 修正建议：
  - 明确约束：关键词预处理统一小写；匹配前统一小写。
  - 添加最小回归用例（如 `IPO/GDP/Fed/S&P`）。

### P1-4 GitHub Actions 缺少并发控制
- 严重性：P1
- 问题描述：15 分钟调度下未定义 `concurrency`，可能并行运行导致 push 冲突。
- 证据：
  - 高频 cron：`finance-news-radar/docs/开发文档.md:613`
  - 直接 commit/push：`finance-news-radar/docs/开发文档.md:647`
- 风险影响：
  - 工作流偶发失败，数据更新中断。
- 修正建议：
  - 在 workflow 增加 `concurrency`（按 workflow 名分组，启用 cancel-in-progress）。
  - 增加 push 冲突重试策略（或使用 rebase + retry）。

## P2（可与开发并行处理）

### P2-1 文档对上游项目行号强依赖，易过期
- 严重性：P2
- 问题描述：附录大量引用“原项目行号”，后续上游变更后会快速失效。
- 证据：
  - `finance-news-radar/docs/开发文档.md:721`
- 风险影响：
  - 维护者误定位代码，增加沟通成本。
- 修正建议：
  - 替换为“函数名 + 模块路径 + commit hash”。

### P2-2 仓库存在 AppleDouble 副文件，影响整洁性
- 严重性：P2
- 问题描述：存在多处 `._*` 文件（macOS 资源分叉副文件）。
- 证据：
  - 示例：`finance-news-radar/docs/._开发文档.md`、`finance-news-radar/scripts/.___init__.py`
- 风险影响：
  - 污染仓库，影响代码评审与跨平台协作。
- 修正建议：
  - 清理已生成副文件。
  - 在 `.gitignore` 添加 `._*`。
