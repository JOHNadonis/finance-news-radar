### 3.3 数据库 Schema

```sql
-- 用户订阅
CREATE TABLE subscriptions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_token TEXT NOT NULL,
  type TEXT NOT NULL CHECK(type IN ('ticker', 'market', 'keyword')),
  value TEXT NOT NULL,
  display_name TEXT,
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(user_token, type, value)
);

-- 模型组（管理员配置的 LLM 后端）
CREATE TABLE model_groups (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,             -- 如 "GPT-4o"
  api_base_url TEXT NOT NULL,     -- 如 "https://api.openai.com/v1"
  api_key TEXT NOT NULL,
  model_name TEXT NOT NULL,       -- 如 "gpt-4o"
  is_default INTEGER DEFAULT 0,
  created_at TEXT DEFAULT (datetime('now'))
);

-- 用户密钥（映射到模型组）
CREATE TABLE access_keys (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  key_value TEXT NOT NULL UNIQUE,  -- "fnr_" + 32位 hex
  label TEXT,
  model_group_id INTEGER NOT NULL REFERENCES model_groups(id) ON DELETE CASCADE,
  is_active INTEGER DEFAULT 1,
  created_at TEXT DEFAULT (datetime('now')),
  last_used_at TEXT
);

-- 管理配置（密码 hash、session 等）
CREATE TABLE admin_config (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

## 4. 功能模块详解

### 4.1 AI 政策分析（流式）

**流程**：
1. 用户点击「刷新分析」→ POST `/api/analysis`
2. 后端读取 `latest-24h.json`，筛选宏观/综合新闻（取前 50 条标题）
3. 构建系统提示词（政策分析师角色 + Mermaid 图表指引）
4. 调用 LLM（`stream: true`），SSE 逐块返回
5. 前端 `useTriggerAnalysis()` 逐块解析，实时显示文字 + 光标动画
6. 流结束后：解析 `---QUESTIONS---` 分隔符提取推荐问题 → 缓存到磁盘 → 发送 meta 事件
7. 下次 GET 请求直接返回缓存

**关键文件**：
- `app/api/analysis/route.ts` — SSE 流式 API
- `hooks/useAnalysis.ts` — `useTriggerAnalysis()` 流式 hook
- `components/dashboard/PolicyAnalysis.tsx` — 流式显示 UI

### 4.2 AI 对话助手（流式）

**流程**：
1. 用户输入 → `useChat.sendMessage()` → POST `/api/chat`
2. 后端构建消息列表：系统提示词 + 上下文（政策分析报告）+ 历史对话
3. 调用 LLM（`stream: true`），SSE 逐块返回
4. 前端逐块更新 assistant 消息内容

**Access Key 支持**：
- `useChat.ts` 从 `localStorage.getItem("fnr_access_key")` 读取
- 有 key 时附加到 fetch body → 后端用 key 对应的模型组配置
- 无 key 时回退到 `llm-config.json`

**引用功能**：
- `TextSelectionQuote.tsx` 监听 `selectionchange`，≥5 字符显示浮动按钮
- 点击后设置 `quotedText` → 打开助手 → 自动拼接引用消息

**关键文件**：
- `app/api/chat/route.ts` — SSE 流式 API + Access Key 分支
- `hooks/useChat.ts` — 流式对话 hook
- `components/assistant/AssistantPanel.tsx` — 完整助手 UI
- `components/assistant/TextSelectionQuote.tsx` — 选中引用

### 4.3 Mermaid 图表

**流程**：
1. LLM 输出 ````mermaid` 代码块
2. `MarkdownContent.tsx` 的 `code` 覆盖检测 `language-mermaid`
3. 渲染 `<MermaidBlock chart={...} />`
4. MermaidBlock 800ms 防抖，内容稳定后才调用 `mermaid.render()`
5. 渲染成功显示 SVG，失败显示原始代码 + 错误信息

**防抖原因**：流式输出时 chart 内容每几毫秒变一次，不防抖会导致反复渲染 SVG → 页面跳动。

**关键文件**：
- `components/assistant/MermaidBlock.tsx` — 懒加载 + 防抖 + 渲染
- `components/assistant/MarkdownContent.tsx` — 代码块分流

### 4.4 后台密钥管理

**管理员流程**：
1. 访问 `/admin` → 输入密码登录
2. 创建模型组（name / base_url / api_key / model_name）
3. 生成 Access Key（选择模型组 + 可选标签）→ 分发给用户

**用户流程**：
1. 在助手设置 Tab 输入 Access Key → 点击「验证」
2. 验证成功 → 存入 localStorage，隐藏手动配置
3. 后续对话自动使用 Access Key 对应的模型

**首次使用**：
- 需设置环境变量 `ADMIN_PASSWORD`
- 首次登录时自动 bcrypt hash 存入 DB
- 后续登录与 DB 中的 hash 比对

**关键文件**：
- `app/admin/page.tsx` — 管理后台 UI
- `app/api/admin/login/route.ts` — 登录（bcrypt + session）
- `app/api/admin/groups/route.ts` — 模型组 CRUD
- `app/api/admin/keys/route.ts` — 密钥 CRUD
- `app/api/models/route.ts` — 公开验证端点
- `lib/admin-auth.ts` — Cookie 验证
- `lib/db.ts` — 数据库操作

## 6. 已知问题与踩坑记录

### 6.1 rsync --delete 会删除 .venv

**问题**：部署时 rsync 用 `--delete` 会删除服务器上的 `.venv` 目录（因为本地没有），导致 Cron Python 脚本全部报 `No such file or directory`，数据停止更新。

**解决**：rsync 必须加 `--exclude .venv`。

**修复命令**（如果已误删）：

### 6.2 Nginx 缓冲 SSE 流

**问题**：Nginx 默认会缓冲代理响应（proxy_buffering），导致 SSE 流式输出变成一次性返回。

**解决**：在 SSE 响应头中添加 `X-Accel-Buffering: no`，告诉 Nginx 不要缓冲该响应。已在 `chat/route.ts` 和 `analysis/route.ts` 中添加。

### 6.3 PM2 restart --update-env 不重读 ecosystem 文件

**问题**：`pm2 restart --update-env` 只刷新 PM2 内存中的环境变量，不会重新读取 `ecosystem.config.cjs` 文件。

**解决**：修改 env 后必须 `pm2 delete fnr-web && pm2 start ecosystem.config.cjs`。

### 6.4 Mermaid 流式输出页面跳动

**问题**：流式输出时 mermaid 代码块每个字符都触发 `mermaid.render()`，反复渲染 SVG 导致页面跳动。

**解决**：MermaidBlock 组件加了 800ms 防抖（`setTimeout`），内容稳定后才渲染。流式期间显示静态占位符。

### 6.5 .venv 中的 macOS symlink 不兼容 Linux

**问题**：如果本地 macOS 的 `.venv` 被 rsync 到 Linux 服务器，里面的 Python symlink 指向 `/opt/homebrew/...`，在 Linux 上不存在。

**解决**：永远不要同步 `.venv`，必须在服务器上用服务器的 Python 创建。

| 方向 | 说明 |
|------|------|
| **对话历史持久化** | 当前对话仅在内存中（刷新即丢失），可存 SQLite 或 localStorage |

| **SSR 缓存** | 首页可加 ISR（增量静态再生成）减少首屏加载时间 |
